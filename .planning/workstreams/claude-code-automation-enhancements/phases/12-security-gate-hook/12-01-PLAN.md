```python
# Current SECURITY_FILES dict (will be moved to security_config.py)
SECURITY_FILES = {
    "core/sandbox.py": "Filesystem boundary enforcement",
    "core/command_filter.py": "Shell command filtering",
    "core/approval_gate.py": "Human-in-the-loop approval",

    "tools/shell.py": "Shell command execution",
    "core/config.py": "Security setting defaults",
    "core/device_auth.py": "Device authentication",
    ".env.example": "Credential patterns and defaults",
}

def check_security_file(file_path):
    """Check if the file is security-critical."""
    for sec_path, description in SECURITY_FILES.items():
        if file_path.endswith(sec_path) or sec_path in file_path:
            return description
    return None
```

Hook protocol:
- Receives JSON on stdin: { hook_event_name, tool_name, tool_input }
- PreToolUse: tool_input has file_path (Write/Edit) or command (Bash)
- Output to stderr is shown to Claude as feedback
- Exit code 0 = allow, exit code 2 = block (PreToolUse only)

Create two new files:

**`.claude/hooks/security_config.py`** -- shared security file registry:
- Move SECURITY_FILES dict here with the expanded 12-file list: the original 10 from security-monitor.py PLUS `.env` (description: "Production secrets and credentials") and `core/encryption.py` (description: "Encryption key management")
- Export a helper function `is_security_file(file_path) -> tuple[bool, str, str]` that returns (is_match, matched_path, description). Uses the same matching logic as the existing `check_security_file()`: checks if `file_path.endswith(sec_path) or sec_path in file_path`
- Add module docstring explaining this is the single source of truth for security-sensitive file definitions, shared by both PreToolUse gate and PostToolUse monitor
- Include `#!/usr/bin/env python3` shebang and keep it dependency-free (stdlib only)

**`.claude/hooks/security-gate.py`** -- PreToolUse blocking hook:
- Shebang: `#!/usr/bin/env python3`
- Module docstring explaining: PreToolUse hook that blocks Write/Edit/Bash operations on security-sensitive files. Exit code 2 = block, exit code 0 = allow.
- Read JSON from stdin (same protocol as security-monitor.py: `json.loads(sys.stdin.read())`)
- Handle JSON parse errors gracefully (exit 0 on failure -- don't block on bad input)
- Extract `tool_name` and `tool_input` from the event
- For **Write** and **Edit** tools (`tool_name in ("Write", "Edit", "write", "edit")`):
  - Get `file_path` from `tool_input`
  - Call `is_security_file(file_path)`
  - If match: print `[security-gate] BLOCKED: {file_path} ({description}) -- approve to continue` to stderr, exit with code 2
  - If no match: exit 0
- For **Bash** tool (`tool_name in ("Bash", "bash")`):
  - Get `command` from `tool_input`
  - Check if the command contains an `rm` or `mv` pattern targeting a security file. Use a regex approach: for each key in SECURITY_FILES, check if the command matches pattern `(rm|mv)\s+.*{escaped_filename}` where `escaped_filename` is the basename or the relative path with dots escaped for regex. Also check for `rm -rf`, `rm -f` variants.
  - If match: print `[security-gate] BLOCKED: Bash command targets security file {matched_path} ({description}) -- approve to continue` to stderr, exit with code 2
  - If no match: exit 0
- For any other tool: exit 0 immediately
- Import SECURITY_FILES from security_config using a sys.path insert for the hooks directory: `sys.path.insert(0, os.path.dirname(os.path.abspath(__file__)))` then `from security_config import SECURITY_FILES, is_security_file`
