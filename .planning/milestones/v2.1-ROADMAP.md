# Milestone v2.1: Multi-Project Workspace

**Status:** SHIPPED 2026-03-26
**Phases:** 1-5
**Total Plans:** 8
**Tests:** 51 workspace-specific (all passing)

## Overview

Tabbed workspace support for the Agent42 IDE page — each tab scopes an independent project with its own file explorer, editor tabs, CC sessions, and terminal, without cross-contamination. Phase 1 locked the server-side registry and client-side namespace conventions; Phase 2 threaded those contracts into all IDE surfaces and rendered the tab bar; Phase 3 added workspace lifecycle management (add, remove, rename). Phases 4-5 were gap closure phases added after milestone audit to fix API wiring and frontend state isolation issues.

## Milestone Summary

**Key Decisions:**
- Namespace isolation (workspace_id on all keys/URIs/API calls) locked in Phase 1 before UI — retrofitting costs 8x more
- Server resolves workspace IDs to paths — never accept raw paths from client (path traversal risk)
- Monaco model swapping (setModel + saveViewState/restoreViewState) over multiple editor instances (80MB RAM each)
- localStorage stale-while-revalidate pattern for workspace tab persistence
- workspace_id moved from standalone query param into Pydantic model body field (Phase 4)
- Always show workspace tab bar (removed <=1 hide guard) — '+' button must always be accessible

**Issues Resolved:**
- File save body/query mismatch on workspace_id (Phase 4)
- Missing workspace_id on search endpoint (Phase 4)
- cc_panel_width and cc_panel_session_id bare localStorage keys not workspace-scoped (Phase 5)
- removeWorkspace reading stale unsaved-files state (Phase 5)

**Issues Deferred:**
- Monaco view state in-memory only — lost on page reload (out of scope for v2.1)

**Technical Debt Incurred:**
- PIPE fallback on Linux/macOS uses Path.home() instead of workspace_path (rare edge case)
- _chat_via_cc sidebar chat uses module-level workspace (intentional — no workspace context in scope)
