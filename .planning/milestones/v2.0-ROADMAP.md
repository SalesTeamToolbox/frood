# v2.0 — MCP Pivot: Agent42 as Claude Code Tool Ecosystem

## Vision

Transform Agent42 from a standalone AI agent platform into an **MCP server ecosystem** that integrates with Claude Code. Users authenticate via their Claude Code subscription (Pro/Max). Agent42 provides the **tooling and environment layer** — memory, security, project management, multi-node control — while Claude Code provides the **intelligence layer**.

## Why

- Eliminates LLM routing complexity (5-layer model router, 9 providers, rate limiting, token tracking)
- Full TOS compliance — just providing tools, not proxying LLM calls
- Leverages Claude Code's superior context management, orchestration, and tool use
- Agent42's memory system (Qdrant + Redis) becomes a genuine upgrade to Claude Code's flat file memory
- Multi-node management (laptop + VPS) is a unique differentiator
- ~40% codebase reduction, ~80% operational complexity reduction

## Phases

### Phase 1: MCP Server Foundation
**Goal:** Agent42 launches as a stdio MCP server that Claude Code can discover and call tools on.

- Create `mcp_server.py` — MCP server using `mcp` Python SDK (stdio transport)
- Add MCP schema generation to `tools/base.py` (Tool → MCP tool definition)
- Create `mcp_registry.py` — adapts ToolRegistry to serve MCP `tools/list` and `tools/call`
- Register a minimal set of tools (filesystem, shell, memory) as proof of concept
- Generate `.mcp.json` for Claude Code integration
- Test: Claude Code can list Agent42 tools and execute them

**Success criteria:** `claude` CLI discovers Agent42 tools and can read/write files through them.

### Phase 2: Full Tool Migration
**Goal:** All 41+ tools are callable as MCP tools with proper security layers.

- Convert all tool categories to MCP-compatible:
  - **Filesystem:** read_file, write_file, grep, diff, linter, outline
  - **Shell:** shell (with CommandFilter), python_exec
  - **Git:** git, pr_generator, repo_map
  - **Memory:** memory_tool (store/recall/search/log)
  - **Project:** app_tool, app_test, project_interview, template, workflow
  - **Web:** browser, http_client, web_search, web_fetch
  - **DevOps:** docker, ssh, tunnel, cron, file_watcher
  - **AI-adjacent:** code_intel, content_analyzer, security_analyzer, dependency_audit
  - **Media:** image_gen, video_gen, vision
  - **Meta:** scoring, behaviour, persona, knowledge, data
- Security layers (sandbox, command_filter, rate_limiter, url_policy) wrap MCP tool execution
- Custom tool plugins (`custom_tools/`) auto-discovered as MCP tools
- Tool extensions (`ToolExtension`) work as MCP tool middleware

**Success criteria:** All 41+ tools work from Claude Code. Security layers enforce same protections.

### Phase 3: Associative Memory System ★ FIRST-CLASS FEATURE
**Goal:** Give Claude Code a human-like long-term memory — relevant memories surface automatically when context triggers them, without being asked. This is Agent42's primary differentiator.

#### 3A: Memory MCP Backend
- Expose memory operations as MCP tools:
  - `agent42_memory_store` — save structured memory with embeddings (Qdrant)
  - `agent42_memory_recall` — semantic search across all stored knowledge
  - `agent42_memory_search` — text/filter search across memories
  - `agent42_memory_context` — get relevant context for current task
  - `agent42_memory_log` — append to conversation history
- Expose memory collections as MCP resources:
  - `memory://entries` — browseable memory entries
  - `memory://sessions/{id}` — session transcripts
  - `memory://projects/{name}` — project-specific knowledge
- Session management: memories persist across Claude Code sessions
- Redis session backend for fast access, Qdrant for semantic search
- Memory categories: solutions, pitfalls, patterns, decisions, preferences, references

#### 3B: Automatic Recall Hook (UserPromptSubmit)
The core innovation — a Claude Code hook that fires on every user prompt:
1. Takes the user's prompt text
2. Generates embedding, queries Qdrant for semantically similar memories
3. Applies relevance filtering:
   - High similarity (>0.85): Always inject — directly relevant
   - Medium similarity (0.70-0.85): Inject if domain/category matches
   - Low similarity (<0.70): Skip — too tangential
4. Respects context budget (max 5 memories, ~2000 tokens)
5. Injects relevant memories as context via stderr (Claude sees them before thinking)
6. Memories are presented as suggestions, not mandates — Claude uses judgment

```
User: "Let's build a secure login for the new app"
                    ↓
Hook fires → semantic search → finds:
  [0.89] "bcrypt hash copy-paste between machines caused encoding issues"
  [0.87] "regenerate hash on server itself, not locally"
  [0.82] "JWT_SECRET must be persistent across restarts"
                    ↓
Claude starts thinking with these memories already in context
```

#### 3C: Automatic Learning Hook (Stop)
Completes the cycle — captures new knowledge automatically:
1. Analyzes what was discussed/built during the conversation turn
2. Extracts learnings: solutions found, pitfalls discovered, patterns established, decisions made
3. Deduplicates against existing memories (don't store what we already know)
4. Generates embeddings and stores in Qdrant with metadata (project, date, category, confidence)
5. Memories strengthen with reuse (confidence score increases when recalled and confirmed)

#### 3D: Memory Lifecycle
- **Decay:** Memories that are never recalled gradually lower in ranking (not deleted)
- **Strengthening:** Memories recalled and confirmed by the user gain confidence
- **Correction:** User can say "that's outdated" or "forget that" → memory flagged/removed
- **Cross-project:** Memories can be scoped to a project or shared globally
- **Cross-node:** Memories from VPS work are available on laptop (via Phase 8 sync)

**Success criteria:**
- Claude automatically surfaces relevant past experiences without being asked
- Working on "auth" naturally triggers auth-related memories from past sessions
- New learnings are captured at session end without manual intervention
- Stale/wrong memories don't pollute context (relevance filtering + decay)
- Feels like working with a colleague who remembers your shared history

### Phase 4: Skills as MCP Prompts
**Goal:** Agent42's 43 skills become MCP prompt templates that Claude Code can discover and use.

- Convert SKILL.md files to MCP prompt definitions
- Skill loader generates MCP `prompts/list` responses
- Skills inject context via MCP `prompts/get` with arguments
- Custom skills in `custom_skills/` auto-discovered as MCP prompts
- Skill enablement managed via dashboard settings

**Success criteria:** Claude Code can list available skills and inject them as context.

### Phase 5: Strip LLM Layer
**Goal:** Remove all LLM intelligence infrastructure. Claude Code is now the brain.

- Remove `providers/` directory entirely
- Remove from `agents/`: agent.py, iteration_engine.py, learner.py, model_catalog.py, model_evaluator.py, model_researcher.py, model_router.py, extension_loader.py
- Remove from `core/`: complexity.py, intent_classifier.py, plan_spec.py, project_spec.py, interview_questions.py, task_queue.py, queue_backend.py, state_manager.py, chat_session_manager.py, capacity.py, rlm_config.py
- Simplify `core/config.py` — remove all LLM-related settings (~40 env vars)
- Simplify `agent42.py` — from orchestrator to MCP server launcher + dashboard starter
- Update all imports and tests
- Remove LLM-specific test files

**Success criteria:** `agent42.py` starts MCP server + dashboard. No LLM API keys needed. Tests pass.

### Phase 6: Dashboard Redesign
**Goal:** Dashboard shifts from "agent control center" to "multi-node environment manager."

- **Remove:** Task queue view, agent dispatch, model routing config, provider health
- **Keep:** Authentication, settings, GitHub accounts, security scanner
- **Add:** Node management view (local + remote nodes, health, connectivity)
- **Add:** Memory browser (browse, search, manage stored memories)
- **Add:** Tool activity log (which tools called, by whom, results)
- **Add:** MCP server status (connected clients, tool registry)
- **Redesign:** Settings page (node config, memory settings, security settings, tool enablement)
- Keep existing CSS/design system, just swap the page content

**Success criteria:** Dashboard shows node health, memory contents, tool activity. No LLM-specific UI remains.

### Phase 7: Remote Node (SSE Transport)
**Goal:** Agent42 runs on a VPS as a remote MCP server accessible from local Claude Code.

- Add SSE/HTTP transport to `mcp_server.py` (in addition to stdio)
- Authentication layer for remote connections (API key + HTTPS)
- SSH tunnel helper for simple secure access
- Tool namespacing: `local_shell` vs `remote_shell` etc.
- Node registration: remote node registers with local dashboard
- `.mcp.json` generation for multi-node setup:
  ```json
  {
    "mcpServers": {
      "agent42-local": {
        "command": "python",
        "args": ["agent42-mcp.py", "--node", "local"]
      },
      "agent42-remote": {
        "command": "ssh",
        "args": ["agent42-prod", "python", "agent42-mcp.py", "--sse"]
      }
    }
  }
  ```
- Remote tool execution with latency-aware timeouts

**Success criteria:** Claude Code can call tools on both local laptop and remote VPS. Dashboard shows both nodes.

### Phase 8: Node Sync
**Goal:** Memory and configuration stay in sync across local and remote nodes.

- Memory sync strategies:
  - **Option A:** Qdrant replication (built-in, requires both nodes running Qdrant)
  - **Option B:** Event-based sync (memory writes publish to both nodes via Redis pub/sub)
  - **Option C:** Pull-on-demand (remote node queries local node's memory when needed)
- Config sync: settings propagate from primary node to secondary
- Event propagation: tool activity on remote node visible in local dashboard
- Conflict resolution for concurrent writes

**Success criteria:** Memory stored on VPS is searchable from laptop. Dashboard shows unified view.

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| MCP SDK immaturity | Pin version, wrap in abstraction layer |
| Claude Code cowork limitations | Start with manual invocation, add cowork patterns later |
| Remote node security | SSH tunnel first, add HTTPS+auth later |
| Memory sync complexity | Start with pull-on-demand (simplest), upgrade to replication if needed |
| Breaking existing users | Create `v2` branch, keep `v1` maintained until migration complete |

## Migration Strategy

1. Create `v2` branch from `dev`
2. Phases 1-3 can be developed incrementally (Agent42 still works as standalone during transition)
3. Phase 5 is the breaking change — after this, Claude Code is required
4. Phases 6-8 are post-migration enhancements
