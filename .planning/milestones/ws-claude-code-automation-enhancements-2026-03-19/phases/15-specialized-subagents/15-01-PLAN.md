Structure:
1. **# Test Coverage Auditor Agent**

2. **## Purpose**: Analyze all source modules for test coverage gaps and produce a prioritized list of untested or under-tested modules ranked by security risk, change frequency, and complexity.

3. **## Context**: Reference the project structure:
   - Source directories: `core/`, `tools/`, `providers/`, `agents/`, `memory/`, `dashboard/`
   - Test directory: `tests/` with `test_<module>.py` naming convention
   - conftest.py provides fixtures: `sandbox`, `command_filter`, `tool_registry`, `mock_tool`
   - Test framework: pytest + pytest-asyncio (auto mode)

4. **## Analysis Steps** (numbered):

   Step 1 - Inventory source modules:
   List all `.py` files in `core/`, `tools/`, `providers/`, `agents/`, `memory/`, `dashboard/` (exclude `__init__.py`, `__pycache__`).

   Step 2 - Inventory test files:
   List all `test_*.py` files in `tests/`.

   Step 3 - Map coverage:
   For each source module, check if a corresponding `tests/test_<module>.py` exists. Mark as COVERED or UNCOVERED. For covered modules, count the number of test functions (grep for `def test_` or `async def test_`).

   Step 4 - Rank by security risk:
   Assign HIGH risk to security-critical modules: `sandbox.py`, `command_filter.py`, `url_policy.py`, `encryption.py`, `device_auth.py`, `config.py`, `auth.py`, `git_auth.py`, `github_oauth.py`, `rate_limiter.py`, `approval_gate.py`.
   Assign MEDIUM risk to modules handling external I/O: API calls, file operations, subprocess execution.
   Assign LOW risk to everything else.

   Step 5 - Rank by change frequency:
   Run `git log --format='' --name-only --since='3 months ago' | sort | uniq -c | sort -rn | head -30` to find most-changed files.

   Step 6 - Rank by complexity:
   Use line count (`wc -l`) as a proxy. Modules over 300 lines = HIGH complexity, 100-300 = MEDIUM, under 100 = LOW.

   Step 7 - Compute priority score:
   Priority = security_risk_weight(3) + change_frequency_weight(2) + complexity_weight(1). Sort descending.

5. **## Output Format**:
   ```
   # Test Coverage Audit Report

   ## Summary
   - Total source modules: N
   - Covered: N (N%)
   - Uncovered: N (N%)

   ## Priority List (Uncovered/Under-tested)

   | Priority | Module | Security Risk | Changes (3mo) | Lines | Test File | Gap |
   |----------|--------|---------------|---------------|-------|-----------|-----|
   | 1 | core/sandbox.py | HIGH | 12 | 450 | test_sandbox.py (3 tests) | Under-tested |

   ## Recommendations
   [Top 5 modules that should get tests first, with rationale]
   ```

   Read `model_router.py`. Find `FREE_ROUTING` dict and any other hardcoded model ID lists. Cross-reference against the OpenRouter API response. Flag dead models in fallback lists.

Structure:
1. **# Migration Impact Agent**

2. **## Purpose**: Trace all usages of a specified package or API across the codebase and flag breaking incompatibilities with file:line references. Used before upgrading a dependency or changing an internal API.

3. **## Context**: Reference the project's import patterns:
   - Python codebase spread across `core/`, `tools/`, `providers/`, `agents/`, `memory/`, `dashboard/`, `tests/`
   - Common dependencies: `openai`, `httpx`, `fastapi`, `pydantic`, `qdrant-client`, `redis`, `bcrypt`, `python-jose`
   - Internal APIs: modules in `core/` are imported by `tools/`, `agents/`, `dashboard/`
   - Entry point: `agent42.py` imports from all major modules

4. **## Usage**: This agent takes an input parameter specifying what to analyze:
   - Package upgrade: e.g., "Upgrading openai from 1.30 to 1.35"
   - API change: e.g., "Renaming project_manager.all_projects() to list_projects()"
   - Module removal: e.g., "Removing core/legacy_router.py"

5. **## Analysis Steps** (numbered):

   Step 1 - Identify the target:
   Parse the input to determine: package name, old API signature, new API signature (if applicable), version range (if applicable).

   Step 2 - Find all import sites:
   Search for `import <package>`, `from <package> import`, and any aliased imports across all `.py` files. Record each with `file:line`.

   Step 3 - Find all usage sites:
   For each imported symbol, grep for its usage in the importing file and any re-exports. Record each call site with `file:line` and the specific usage pattern (function call, attribute access, class instantiation, etc.).

   Step 4 - Analyze breaking changes:
   If upgrading a package: fetch the changelog or migration guide (from PyPI, GitHub releases, or docs). Compare each usage site against known breaking changes.
   If changing an internal API: compare each call site against the new signature. Check argument order, removed parameters, type changes.

   Step 5 - Assess test coverage of affected sites:
   For each affected file, check if a corresponding test exists in `tests/`. If yes, check if the tests exercise the affected code paths. Flag untested migration risks as HIGH priority.

   Step 6 - Generate migration plan:
   For each breaking change, specify the exact fix needed at each file:line.

   ## Migration Steps (ordered)
   1. [Step with file:line reference]
   2. [Step with file:line reference]

Create `.claude/agents/deploy-verifier.md` following the existing agent pattern.

Structure:
1. **# Deploy Verifier Agent**

2. **## Purpose**: Run pre-deployment validation checks to catch configuration errors, missing files, broken imports, and environment issues before they crash the production server.

3. **## Context**: Reference the deployment architecture:
   - Entry point: `agent42.py` (imports from all major modules)
   - Config: `core/config.py` Settings class with `from_env()` loading from os.getenv
   - Env template: `.env.example` documents all expected environment variables
   - Production: systemd service, `~/agent42/` on Contabo VPS
   - Known pitfall (94): new modules not committed cause `ModuleNotFoundError` on production

4. **## Pre-Deploy Checks** (numbered):

   Check 1 - Import verification:
   Run `python -c "import agent42"` (or the equivalent that imports the main entry point). If this fails, report the ImportError with the missing module name. Then check if that module file exists but is untracked in git (the Pitfall 94 scenario).

   Check 2 - Environment variable validation:
   Read `core/config.py` and extract all `os.getenv()` calls. Read `.env.example` and extract all documented variables. Cross-reference:
   - Variables in code but not in `.env.example` = undocumented (WARN)
   - Variables in `.env.example` but not in code = stale documentation (INFO)
   - Required variables (no default or default is None) without `.env.example` entry = HIGH risk

   Check 3 - Method signature matching:
   Grep for common cross-module call patterns that have caused issues:
   - `project_manager.` calls in `server.py` — verify methods exist on ProjectManager
   - `skill.enabled` or `skill.` attribute access — verify against Skill class
   - `model_evaluator.` and `model_catalog.` calls — verify against actual classes
   Run `grep -rn "\.all_projects\|\.enabled\b" core/ dashboard/ tools/` and verify each call target exists.

   Check 4 - Untracked required files:
   Run `git status --porcelain` to find untracked `.py` files. For each untracked file, check if any tracked file imports it. If yes, flag as CRITICAL — this file will be missing after `git pull` on production.

   Check 5 - Requirements consistency:
   Run `pip freeze` and compare against `requirements.txt`. Flag packages that are installed but not in requirements (potential undeclared dependency) and packages in requirements but not installed (potential missing dep on fresh deploy).

   Check 6 - Configuration consistency:
   Check for common deployment misconfigurations:
   - `QDRANT_URL` using Docker hostname (`qdrant:6333`) on bare metal (Pitfall 115)
   - `SANDBOX_ENABLED=false` in production (Pitfall 106)
   - `DASHBOARD_HOST=0.0.0.0` without firewall mention
   Read `.env` (if accessible) or `.env.example` for these patterns.

5. **## Output Format**:
   ```
   # Deploy Verification Report

   ## Summary
   - Checks run: 6
   - PASS: N | WARN: N | FAIL: N

   ## Results

   | Check | Status | Details |
   |-------|--------|---------|
   | Import verification | PASS/FAIL | [details] |
   | Env var validation | PASS/WARN | [N documented, N undocumented] |
   | Method signatures | PASS/FAIL | [details] |
   | Untracked files | PASS/FAIL | [N untracked imports found] |
   | Requirements | PASS/WARN | [details] |
   | Configuration | PASS/WARN | [details] |

   ## Critical Issues (must fix before deploy)
   [List with file:line references]

   ## Warnings (review recommended)
   [List with file:line references]

   ## Deploy Readiness: READY / NOT READY
   ```
