- name: prod-check
- description: Run all production health checks via SSH in one pass
- disable-model-invocation: true

The skill instructions must contain:

1. **Prerequisites section**: Document that SSH config alias `agent42-prod` must be configured (Contabo VPS, port 2222, user `deploy`). Include a verification step: run `ssh agent42-prod "echo OK"` before proceeding. If this fails, instruct Claude to tell the developer their SSH config needs setup and stop.

2. **Six numbered health checks**, each with a specific SSH command to run via `Bash` tool and what to report:

   Check 1 - Systemd Service Status:
   `ssh agent42-prod "sudo systemctl status agent42 --no-pager -l"`
   Report: active/inactive, uptime, recent restarts, memory usage if shown

   Check 2 - Recent Logs (last 50 lines):
   `ssh agent42-prod "tail -50 ~/agent42/agent42.log"`
   Report: count of ERROR/WARNING lines, any stack traces, last activity timestamp

   Check 3 - Qdrant Health:
   `ssh agent42-prod "curl -s http://localhost:6333/healthz"`
   Report: healthy/unhealthy, response content

   Check 4 - Redis Ping:
   `ssh agent42-prod "redis-cli ping"`
   Report: PONG = healthy, anything else = problem

   Check 5 - Dashboard HTTP Check:
   `ssh agent42-prod "curl -s -o /dev/null -w '%{http_code}' http://localhost:8000/api/health"`
   Report: 200 = healthy, other codes = problem (include the actual status code)

   Check 6 - Disk Usage:
   `ssh agent42-prod "df -h / /var/lib/qdrant"`
   Report: usage percentage for each mount, warn if any > 80%

3. **Output format**: After running all checks, summarize in a markdown table:
   ```
   | Check | Status | Details |
   |-------|--------|---------|
   | Agent42 Service | OK/WARN/FAIL | [specifics] |
   | Logs | OK/WARN/FAIL | [specifics] |
   | Qdrant | OK/WARN/FAIL | [specifics] |
   | Redis | OK/WARN/FAIL | [specifics] |
   | Dashboard | OK/WARN/FAIL | [specifics] |
   | Disk | OK/WARN/FAIL | [specifics] |
   ```

4. **Overall assessment**: After the table, provide an overall verdict (ALL HEALTHY, DEGRADED, or CRITICAL) with any recommended actions.

Do NOT use `!` backtick syntax to run commands at skill load time -- each check should be a separate Bash tool call so Claude can analyze results step-by-step. Do NOT make this skill auto-invocable (disable-model-invocation must be true). Do NOT create any Python files -- this is a pure instruction skill.

Create `.claude/skills/add-pitfall/` directory and `SKILL.md` file.

YAML frontmatter:
- name: add-pitfall
- description: Auto-format and auto-number a new entry in the CLAUDE.md pitfalls table
- disable-model-invocation: true

The skill instructions must contain:

1. **Input gathering**: If arguments were provided with the slash command, parse them as `[area] [pitfall description] [correct pattern]`. If no arguments or incomplete arguments, ask the developer for:
   - Area (e.g., Deploy, Auth, Dashboard, Config, Tools, Apps, Routing, Memory, etc.)
   - Pitfall -- what goes wrong (one sentence)
   - Correct Pattern -- how to do it right (one sentence)

2. **Step: Read current state**: Read `CLAUDE.md` and find the last entry number in the Common Pitfalls table. Search for lines matching the pattern `| NNN |` where NNN is a number. Extract the highest number found. The new entry number = highest number + 1. Do NOT hardcode any number -- always detect dynamically.

3. **Step: Format the entry**: Build the new table row as:
   `| {next_number} | {Area} | {Pitfall} | {Correct Pattern} |`
   Ensure pipe characters within the pitfall or pattern text are escaped or avoided. Keep each cell to one sentence for readability.

4. **Step: Insert the entry**: Use the Edit tool to insert the new row at the END of the pitfall table, BEFORE the blank line and `---` separator that follows the last table row. The insertion point is immediately after the last `| NNN | ... |` row. Do NOT insert after the `---` separator. Do NOT insert inside the "Extended Reference" section.

5. **Step: Verify**: Read the modified section of CLAUDE.md to confirm:
   - The new entry number is correct (last + 1)
   - The row has exactly 4 pipe-separated columns
   - The table still renders as valid markdown
   - The "Extended Reference" section below is undisturbed

6. **What NOT to do**: Do NOT hardcode the next pitfall number (always detect dynamically). Do NOT modify any other section of CLAUDE.md. Do NOT delete or move existing pitfall entries. Do NOT insert the new row after the `---` separator. Do NOT create any Python files.
